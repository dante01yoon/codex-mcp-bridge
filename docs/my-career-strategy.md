## 내가 가진 것의 정체

8년간 프론트엔드를 했으면 React나 Vue를 잘 쓰는 것이 자산이 아니다. 그건 AI가 이미 한다.

진짜 자산은 세 가지다:

### 1. 사용자 감각

"사용자가 여기서 헤맬 거다"를 코드 한 줄 쓰기 전에 아는 감각. 8년 동안 실제 사용자가 만든 것을 부수고, 오용하고, 예상과 다르게 쓰는 걸 반복해서 보면서 생긴 것. 교육으로 배울 수 없고, AI가 갖고 있지 않다.

### 2. 복잡한 상태 시뮬레이션

"이 상태에서 사용자가 뒤로가기를 누르고, 네트워크가 끊기고, 다시 접속하면 어떻게 되지?" 이런 사고를 자연스럽게 하는 사람이 8년차다.

### 3. "만들면 안 된다"는 판단

주니어는 시키는 걸 다 만든다. 8년차는 "이 기능은 사용자를 오히려 혼란스럽게 한다, 빼야 한다"를 말할 수 있다.

## 위험한 것과 위험하지 않은 것

```
위험한 것:
  "피그마 보고 React 컴포넌트 만드는 일"
  → v0, Bolt, Lovable이 이미 자동화하고 있고, 가속화될 것

위험하지 않은 것:
  "이 제품이 사용자에게 어떤 경험이어야 하는지 결정하는 일"
  → 이건 더 중요해지고 있음
```

8년 경력이 위험한 쪽에만 서 있다면 문제다. 하지만 8년차면 이미 후자 쪽에 상당 부분 발을 딛고 있다.

## 방향: AI 에이전트의 얼굴을 만드는 사람

### 왜 이 방향인가

AI 에이전트 생태계에서 가장 부족한 것이 **좋은 인터페이스**다.

모든 에이전트는 결국 사람과 만나는 지점이 필요하다:
- 에이전트가 중간에 판단을 요청하는 화면 (human-in-the-loop)
- 여러 에이전트가 동시에 작업하는 상태를 보여주는 대시보드
- 에이전트의 결과를 사람이 검토하고 승인하는 워크플로우
- 스트리밍으로 실시간 생성되는 AI 응답의 UI 패턴

이걸 제대로 만들 수 있는 사람은 **프론트엔드를 깊이 알면서 동시에 에이전트가 어떻게 동작하는지 이해하는 사람**이다. 이 교차점에 있는 사람이 지금 극히 적다.

### 왜 나인가

```
AI 엔지니어 → 백엔드에서 에이전트를 돌리는 데 집중, UI는 후순위
디자이너    → 기술적 제약을 모름, 에이전트 동작 방식 이해 부족
나          → 프론트엔드 8년 + MCP 프로토콜 이해 + 에이전트 아키텍처 파악
             = 이 빈 자리를 채울 수 있는 위치
```

codex-mcp-bridge를 만들면서 MCP 프로토콜을 이해하고, 서브에이전트와 팀의 구조를 파악한 것이 출발점이다. 프론트엔드만 하는 8년차가 아니라, **에이전트 생태계의 구조를 이해하는 프론트엔드 8년차**라는 것이 차별점이다.

## 구체적 행동 계획

### 원칙: 지금 있는 곳에서 한 발짝

```
X: 프론트엔드를 버리고 AI 엔지니어가 되겠다
O: 프론트엔드 위에 에이전트 생태계의 이해를 얹겠다
```

8년을 투자한 영역을 떠나서 0년차로 새 영역에 들어가는 것은 자산을 전부 매각하고 모르는 시장에 올인하는 것과 같다.

### 원칙: 생계를 유지하면서 전환

```
시간의 80%: 현재 수입을 유지하는 일
시간의 20%: 다음 단계를 준비하는 일 (MCP Playground 등)
```

직장을 그만두고 6개월 부트캠프를 다니지 않는다. 20%에서 작은 실험을 반복하고, 그 실험이 쌓여서 포트폴리오와 평판이 된다.

### 원칙: "무엇을 배울까"보다 "무엇을 만들까"

튜토리얼 지옥에 빠지지 않는다. **"내 업무/관심사에서 지금 고통스러운 문제가 뭐지?"부터 시작한다.** 문제가 학습을 이끌게 해야지, 학습이 문제를 찾아다니면 안 된다.

→ 그 첫 번째 문제: "MCP 서버를 테스트하는 경험이 너무 나쁘다" → MCP Playground

## 기술 너머의 것

> 이 섹션은 "시장에서 유리한 위치"가 아니라 "후회 없는 삶"에 대한 것이다.

### 특정 기술은 변하지만, 이것들은 변하지 않는다

```
호기심에서 움직이기
  "뒤처지지 않기 위해"가 아니라
  "이거 내 일에 써먹으면 재밌겠는데"라는 동기

"왜"를 묻는 습관
  "코딩을 배워라"가 아니라
  "이게 왜 이렇게 되는 거지?"라는 질문
  → 구조를 이해하려는 습관은 도구가 바뀌어도 이전됨

충분함의 감각
  "이 정도면 충분하다"를 아는 것
  → 돈에 쫓기는 사람은 판단력을 잃고,
    판단력을 잃은 사람은 어떤 시장에서든 취약함

실패에 대한 내성
  AI 시대의 가장 큰 위험은 실직이 아니라
  "내가 할 수 있는 것이 아무것도 없다"는 무력감
  → 작은 실패를 반복하고 다시 일어나는 경험이 이것에 저항하게 함
```

### Jack Clark (Anthropic 공동창업자)의 말

> "인간을 차별화할 것은 AI 활용을 위한 특정 기술이 아니라,
> 높은 수준의 호기심과 주체성(agency)을 갖추는 것이다."

### 이 문서의 전략과의 관계

위의 것들은 전략 문서를 부정하는 것이 아니다. 시장 전략은 유용하다. 하지만 시장 전략만으로 삶을 설계하면, 자신을 시장의 부품으로 최적화하게 된다.

**품위 있는 사람은 시장에 적응하는 사람이 아니라, 시장이 무엇이든 자기 기준을 가진 사람이다.**

## 첫 번째 결과물: MCP Playground

### 왜 이것인가

```
내 상황에서 가장 자연스러운 프로젝트:
  - codex-mcp-bridge를 만들면서 MCP 생태계를 이미 이해
  - "MCP 서버 테스트 경험이 나쁘다"는 실제 고통을 겪음
  - 프론트엔드 기술이 핵심인 프로젝트
  - 경쟁자가 약하거나 부재 (Hoot: 14 stars, Inspector: 저장 불가)
```

### 포지셔닝

> "MCP Inspector가 curl이라면, MCP Playground는 Postman이다
> — 단, 오픈소스이고 MCP에 특화되어 있다."

### MVP 핵심 기능 7가지

1. **서버 연결** — stdio/SSE/HTTP, 프로필 저장, MCP JSON 붙여넣기
2. **도구 탐색** — JSON Schema → 시각적 폼 자동 생성
3. **도구 실행** — 폼 + Raw JSON 입력, 결과 포맷팅
4. **요청 히스토리** — 모든 호출 자동 기록, 재실행
5. **컬렉션** — 요청을 폴더로 정리, 일괄 실행 (Run All)
6. **환경변수** — key-value 환경 정의, {{변수}} 치환
7. **결과 비교** — 이전 실행과 diff

## 정량적 결과물로 만드는 법

### 두 종류의 정량 지표

```
1. 오픈소스 지표 (포트폴리오/평판)
   - GitHub stars
   - npm weekly downloads
   - 컨트리뷰터 수
   - 블로그 포스트 조회수
   → "AI 에이전트 인터페이스 전문가"라는 평판

2. 매출 지표 (사업화, 후순위)
   - 유료 사용자 수
   - MRR (월 반복 매출)
   → 오픈소스 코어 + 유료 기능(SaaS)
```

첫 단계에서는 1번에 집중한다. 평판이 먼저이고, 매출은 평판 위에 올린다.

### 선보이는 곳

```
즉시:
  - GitHub 프로필 핀 프로젝트
  - 이력서/LinkedIn: "MCP Playground 제작, GitHub ★ X개, npm 주간 다운로드 Y회"
  - 기술 블로그: 제작 과정과 기술적 의사결정 기록

중기:
  - 밋업/컨퍼런스 발표 (FEConf, JSConf Korea, 온라인 글로벌)
  - AI 에이전트 커뮤니티 인지도
  - 도구를 사용하는 회사/개인으로부터의 인바운드

장기:
  - "AI 에이전트 인터페이스" 키워드에 내 이름이 연결
  - 채용이든 사업이든, 기회가 찾아오는 구조
```
